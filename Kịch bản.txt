CÔNG NGHỆ SỬ DỤNG
─────────────────────────────────────────────────────────────────
"Dự án được chia làm hai phần rõ ràng là Frontend và Backend."

► FRONTEND – giao diện người dùng:
  • Next.js 14 (App Router) + React 18 + TypeScript
    → Framework chính, hỗ trợ SSR và routing theo thư mục.
  • Tailwind CSS → tạo giao diện responsive mà không cần viết CSS thủ công.
  • Zustand → quản lý state toàn cục (giỏ hàng, auth, yêu thích)
    thay thế Redux vì nhẹ hơn và không cần boilerplate.
  • Firebase SDK (client) → xác thực người dùng phía trình duyệt.
  • Axios → gửi HTTP request tới backend, có cấu hình interceptor tự
    động đính kèm token và refresh khi hết hạn.
  • Framer Motion → hiệu ứng animation trên trang Generate.
  • React Hot Toast → thông báo nhanh cho người dùng.

► BACKEND – máy chủ:
  • Node.js + Express.js → máy chủ REST API.
  • Firebase Admin SDK → xác thực token phía server, đọc/ghi Firestore,
    lưu trữ file trên Firebase Storage.
  • Google AI Studio API (Gemini 2.5 Flash Image) → sinh ảnh tranh tô màu.
  • Google AI Studio API (Gemini 2.5 Flash Text) → chatbot tư vấn.
  • Helmet → bảo mật HTTP headers.
  • Morgan → ghi log request.
  • Express Rate Limit → giới hạn 100 request mỗi 15 phút mỗi IP.
  • Joi → validate dữ liệu đầu vào.
  • Nodemailer → gửi email đặt lại mật khẩu.

► DATABASE & STORAGE:
  • Firestore (NoSQL) → lưu toàn bộ dữ liệu: sản phẩm, đơn hàng,
    người dùng, kết quả generate AI.
  • Firebase Storage → lưu file ảnh (sản phẩm + ảnh AI tạo ra).

► CẤU TRÚC PROJECT (Controllers, Services, Entities):
  + Backend:
    - Entities: 5 đối tượng dữ liệu chính (Users, Products, Orders, Generations, Settings) lưu trên Firestore.
    - Controllers: 14 file Routes tiếp nhận và điều hướng API (Auth, AI, Thanh toán...).
    - Services: 6 file xử lý logic nghiệp vụ lõi.
    - Actions: Hàng chục API thao tác thực thi (CRUD, tạo link VNPay/MoMo, sinh ảnh).
  + Frontend (Next.js): Hàng chục Server/Client Actions, Hooks và Utils để gọi API.

  [4] Tạo tranh AI

  [5] Quản trị
      Admin đăng nhập → dashboard quản lý sản phẩm, đơn hàng, người dùng.
      Hai lớp xác thực admin đảm bảo bảo mật.

CÁC CHỨC NĂNG CHÍNH
─────────────────────────────────────────────────────────────────
"Hệ thống có 8 nhóm chức năng chính:"

  1. Xác thực người dùng
     Đăng ký, đăng nhập, đăng xuất qua Firebase Auth.
     Hỗ trợ "Ghi nhớ đăng nhập": nếu bật → lưu token vào localStorage,
     nếu tắt → lưu vào sessionStorage, tự xóa khi đóng trình duyệt.
     Đặt lại mật khẩu: backend gửi email qua Nodemailer với link reset.

  2. Sản phẩm
     Danh sách tranh tô màu thực tế với lọc theo danh mục, độ khó, giá.
     Phân trang.
     Admin có thể thêm / sửa / xóa / ẩn sản phẩm, upload ảnh.

  3. Giỏ hàng
     Zustand + persist (localStorage) → giỏ hàng không mất khi F5.
     Hỗ trợ "Lưu để mua sau", chọn/bỏ chọn từng sản phẩm.
     Các voucher: YULING10 (giảm 10%), YULING20 (20%), GIAMGIA15 (15%).

  4. Thanh toán đa phương thức
     VNPay → tạo URL với chữ ký HMAC-SHA512.
     MoMo  → tạo URL với chữ ký HMAC-SHA256.
     COD (tiền mặt khi nhận hàng).
     Chuyển khoản ngân hàng.
     Sau thanh toán online → callback xác minh chữ ký trước khi cập nhật đơn.

  5. Đơn hàng
     Vòng đời đơn hàng: pending → processing → shipping → delivered / cancelled.
     Người dùng xem lịch sử đơn. Admin cập nhật trạng thái.

  6. Tạo tranh AI
     Tính năng đặc biệt nhất – sẽ trình bày chi tiết ở phần tiếp theo.

  7. Chatbot AI
     Chatbot nổi trên mọi trang, gọi Gemini 2.5 Flash text.
     Mỗi lần gọi AI, backend tự động lấy danh sách sản phẩm đang active
     từ Firestore (tối đa 30) rồi nhúng vào system prompt → AI biết
     chính xác sản phẩm nào đang bán, giá bao nhiêu, độ khó thế nào.
     Duy trì lịch sử hội thoại (20 tin nhắn gần nhất) → AI nhớ ngữ cảnh.

  8. Admin Dashboard
     Quản lý đầy đủ: sản phẩm, đơn hàng, người dùng, cài đặt hệ thống.

─────────────────────────────────────────────────────────────────
KIẾN TRÚC BACKEND
─────────────────────────────────────────────────────────────────
"Backend được tổ chức theo mô hình phân lớp rõ ràng:"

  Routes  →  Middleware  →  Services  →  Firebase (DB/Storage)

  • Routes: nhận request, validate đầu vào cơ bản, gọi service.
  • Middleware:
    - auth.js: xác thực Firebase ID Token, gắn req.user.
    - adminAuth.js: hai lớp kiểm tra quyền admin (email list + Firestore role).
  • Services: chứa toàn bộ business logic
    (productService, orderService, paymentService, chatService, v.v.)
  • Config: khởi tạo Firebase Admin SDK một lần duy nhất.
  • Utils: uploadToStorage, deleteFromStorage, helpers.

NHỮNG NỖ LỰC CẢI THIỆN CODE
─────────────────────────────────────────────────────────────────
"Trong quá trình phát triển, em đã chú trọng cải thiện chất lượng code
theo nhiều hướng:"

  ► Bảo mật
    - Helmet: tự động thiết lập các HTTP security headers (X-Frame-Options,
      Content-Security-Policy, v.v.) chống XSS, clickjacking.
    - CORS: chỉ cho phép domain frontend cụ thể, không mở wildcard.
    - Rate limiting: giới hạn 100 request/15 phút mỗi IP → phòng brute-force.
    - Hai lớp admin auth: lớp 1 kiểm tra email list (nhanh, không cần DB),
      lớp 2 kiểm tra role trong Firestore (chính xác, chống giả mạo).
    - Validate input: Joi validate tất cả request body trước khi xử lý.
    - Giới hạn prompt AI: tối đa 500 ký tự, lọc nội dung 18+.
    - Xác minh chữ ký HMAC sau callback thanh toán trước khi cập nhật DB.
    - Kiểm tra ownership: chỉ chủ sở hữu mới xem được kết quả generate.

  ► Hiệu năng & trải nghiệm người dùng
    - Pattern 202 Accepted + Polling: AI tạo ảnh có thể mất 30-120 giây.
      Nếu dùng request đồng bộ, client sẽ bị treo timeout. Giải pháp:
      trả về 202 ngay, frontend poll mỗi 5 giây, tối đa 5 phút.
      Polling dùng ref (useRef) để tránh memory leak khi unmount.
    - Axios interceptor: tự động lấy Firebase token mới (getIdToken(true))
      trước mỗi request → không bao giờ gửi token hết hạn.
      Nếu server trả 401 → tự refresh một lần → thất bại → redirect login.
    - SSR-safe storage: Next.js chạy cả ở server (Node) lẫn trình duyệt.
      safeStorage.ts bọc localStorage/sessionStorage trong try-catch,
      kiểm tra typeof window → không crash khi render phía server.
    - Zustand persist: giỏ hàng, yêu thích lưu localStorage → F5 không mất data.
    - Chatbot context: inject live product data mỗi lần gọi AI → luôn cập nhật,
      không cần fine-tune hay cập nhật model thủ công.

  ► Khả năng bảo trì
    - Tách biệt hoàn toàn Business Logic vào Services → Routes chỉ gọi service.
    - Prompt template dùng placeholder ({{USER_PROMPT}}, {{STYLE}},
      {{COMPLEXITY}}) → dễ chỉnh sửa hướng dẫn AI mà không đụng logic.
    - Firebase config hỗ trợ cả hai định dạng private key (base64 encode
      hoặc \n-escaped) → deploy dễ dàng trên nhiều platform khác nhau.

─────────────────────────────────────────────────────────────────
LUỒNG XỬ LÝ CHI TIẾT (7 BƯỚC)
─────────────────────────────────────────────────────────────────

  BƯỚC 1 – NGƯỜI DÙNG GỬI YÊU CẦU
  ─────────────────────────────────
  Người dùng phải đăng nhập (route được bảo vệ bởi authenticateUser).
  Nhập mô tả (prompt) tối đa 500 ký tự, chọn:
    • Độ phức tạp: easy / medium / hard (ảnh hưởng chi tiết của tranh)
    • Style mặc định: realistic
  Frontend gửi POST /api/generate/paint-by-numbers kèm Bearer Token.

  BƯỚC 2 – BACKEND NHẬN VÀ TẠO RECORD NGAY LẬP TỨC
  ───────────────────────────────────────────────────
  Vì AI tạo ảnh có thể mất 30–120 giây, nếu giữ connection mở sẽ timeout.
  Giải pháp: backend tạo ngay một document trong Firestore collection
  "generations" với status = "processing", rồi trả về HTTP 202 Accepted
  kèm generationId trong vòng ~100ms.

  Dữ liệu lưu Firestore:
    { id, userId, prompt, style, complexity,
      status: "processing", imageUrl: "", createdAt }

  → Frontend nhận được generationId và bắt đầu polling.

  BƯỚC 3 – XÂY DỰNG PROMPT ENGINEERING
  ──────────────────────────────────────
  Sau khi trả 202, backend chạy ngầm hàm generatePaintByNumbers() mà
  KHÔNG await (fire-and-forget pattern) → không block Event Loop.

  Hàm buildLineArtPrompt() thay thế ba placeholder trong template:
    {{USER_PROMPT}}  → mô tả của người dùng
    {{STYLE}}        → phong cách (mặc định: realistic)
    {{COMPLEXITY}}   → easy / medium / hard

  Template được thiết kế như một "vai diễn" (persona prompt):
    - Giao vai: "You are a professional paint-by-numbers illustrator"
    - Yêu cầu cứng: line-art đen trắng, vùng đóng kín, đánh số,
      bảng màu bên dưới dạng ô màu thực tế (không in tên màu),
      không watermark, không nội dung 18+.
    - Yêu cầu chất lượng: high clarity, print-ready.

  Tại sao dùng Prompt Engineering thay vì train model riêng?
  → Model Gemini đã được train sẵn với lượng dữ liệu khổng lồ.
    Thay vì tốn tài nguyên train từ đầu, em "lái" model bằng prompt
    chi tiết → tiết kiệm tài nguyên, dễ chỉnh sửa, kết quả tốt.

  BƯỚC 4 – GỌI GOOGLE AI STUDIO API
  ────────────────────────────────────
  URL: https://generativelanguage.googleapis.com/v1beta/
       models/gemini-2.5-flash-image:generateContent?key=<API_KEY>

  Request body:
    {
      "contents": [{ "parts": [{ "text": "<prompt đã build>" }] }],
      "generationConfig": {
        "responseModalities": ["TEXT", "IMAGE"]
      }
    }

  Điểm quan trọng:
    • responseModalities: ["TEXT", "IMAGE"] → yêu cầu AI trả về cả ảnh.
    • Timeout: 120 giây (vì sinh ảnh chất lượng cao tốn thời gian).
    • API Key lưu trong biến môi trường (.env), KHÔNG hardcode trong code.

  Response từ Google AI:
    {
      "candidates": [{
        "content": {
          "parts": [
            { "text": "..." },
            { "inlineData": {
                "mimeType": "image/png",
                "data": "<base64 string>" }}
          ]
        }
      }]
    }

  Code trích xuất dữ liệu ảnh:
    → Duyệt qua parts[], tìm phần tử có inlineData.
    → Lấy chuỗi base64 → chuyển thành Node.js Buffer nhị phân:
       Buffer.from(base64Data, "base64")

  BƯỚC 5 – UPLOAD LÊN FIREBASE STORAGE
  ──────────────────────────────────────
  Hàm uploadToStorage(buffer, fileName, folder) trong storageHelpers.js:
    1. Tạo tham chiếu file: storage.bucket().file("generations/<fileName>")
    2. Dùng file.save(buffer, { metadata: { contentType: "image/png" }})
    3. Gọi file.makePublic() → file truy cập được bằng URL công khai
    4. Trả về URL dạng:
       https://storage.googleapis.com/<bucket>/generations/<fileName>

  Tại sao lưu Storage thay vì trả base64 thẳng?
    • Chuỗi base64 ảnh PNG ~500KB → 600KB text → nặng khi truyền qua API.
    • URL ngắn gọn, frontend chỉ cần <img src="url"> là xong.
    • Lưu lâu dài, người dùng có thể xem lại sau.

  BƯỚC 6 – CẬP NHẬT FIRESTORE VÀ THÔNG BÁO HOÀN THÀNH
  ──────────────────────────────────────────────────────
  Sau khi có URL, backend cập nhật document Firestore:
    { status: "completed", imageUrl: "<url>", completedAt: "..." }

  Nếu bất kỳ bước nào lỗi (API lỗi, Storage lỗi, v.v.):
    { status: "failed", error: "<message>", failedAt: "..." }

  BƯỚC 7 – FRONTEND POLLING VÀ HIỂN THỊ KẾT QUẢ
  ─────────────────────────────────────────────────
  Frontend dùng setInterval mỗi 5 giây gọi:
    GET /api/generate/status/:generationId

  Backend kiểm tra: userId trong Firestore phải khớp req.user.uid
  → Bảo mật: người khác không thể xem kết quả của người khác.

  Logic polling:
    • Nếu status = "processing" → tiếp tục chờ.
    • Nếu status = "completed"  → xóa interval, hiển thị ảnh.
    • Nếu status = "failed"     → xóa interval, hiển thị lỗi.
    • Tối đa 60 lần × 5 giây = 5 phút → nếu quá → báo timeout.

  Kỹ thuật tránh memory leak:
    → Lưu ID của setInterval vào useRef.
    → Trong useEffect return cleanup → clearInterval khi component unmount.
    → Đảm bảo không tiếp tục gọi API sau khi người dùng thoát trang.

  Sau khi có ảnh, người dùng có thể:
    • Tải xuống (fetch blob → tạo <a> tag programmatically → click).
    • Thêm vào giỏ hàng như sản phẩm AI.
    • Yêu thích (lưu vào favoriteStore).

  • API đơn giản: gọi HTTP thuần, không cần SDK nặng.
  • Chất lượng đủ tốt cho usecase tranh tô màu theo số.
  • Nếu model này lỗi, code chuẩn bị sẵn danh sách GOOGLE_IMAGE_MODELS[]
    để fallback (hiện chỉ dùng 1 model, dễ mở rộng).

PHẦN 4 – DỰ ĐOÁN CÂU HỎI GIÁO VIÊN & HƯỚNG TRẢ LỜI
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

══════════════════ NHÓM 1: CÂU HỎI VỀ AI ══════════════════

❓ Q1: "Em có train model AI không? Nếu không thì có gọi là AI không?"
✅ Trả lời:
  Dạ thưa thầy/cô, em không tự train model vì mục tiêu của đồ án là
  xây dựng ứng dụng thực tế, không phải nghiên cứu model.
  Em sử dụng API của Google Gemini – đây là Large Language Model /
  Multimodal Model đã được Google train sẵn trên hàng tỷ dữ liệu.
  Việc tích hợp AI vào sản phẩm thực tế thông qua API là cách tiếp cận
  phổ biến trong công nghiệp (như ChatGPT API, Midjourney API v.v.).
  Điểm em đóng góp là thiết kế Prompt Engineering - cách "lái" AI
  cho ra đúng định dạng tranh tô màu theo số với các yêu cầu kỹ thuật
  cụ thể, và xây dựng toàn bộ hệ thống xử lý kết quả xung quanh nó.

❓ Q2: "Prompt Engineering là gì? Em hiểu nó thế nào?"
✅ Trả lời:
  Prompt Engineering là nghệ thuật thiết kế câu lệnh (prompt) để điều
  khiển hành vi của AI model, giống như "lập trình bằng ngôn ngữ tự nhiên".
  Trong dự án, em thiết kế prompt theo cấu trúc:
    1. Giao vai trò: "You are a professional paint-by-numbers illustrator"
       → AI "suy nghĩ" theo góc nhìn của chuyên gia vẽ tranh.
    2. Yêu cầu cứng (hard requirements): đóng vùng kín, đánh số,
       bảng màu bên dưới, không tên màu, không nội dung 18+.
    3. Biến động: {{USER_PROMPT}}, {{STYLE}}, {{COMPLEXITY}} được
       thay thế runtime theo lựa chọn của người dùng.
  Kết quả: cùng một model nhưng luôn cho ra tranh tô màu theo số
  đúng định dạng, không cho ra ảnh tô màu tùy ý hay ảnh thông thường.

❓ Q3: "responseModalities là gì? Tại sao cần nó?"
✅ Trả lời:
  responseModalities là tham số yêu cầu Google Gemini trả về LOẠI dữ liệu
  nào trong response. Mặc định Gemini chỉ trả về text.
  Khi em đặt responseModalities: ["TEXT", "IMAGE"], model hiểu rằng
  ngoài text giải thích còn phải trả về dữ liệu ảnh dạng inlineData
  (Base64 encoded).
  Nếu không có tham số này → chỉ nhận được text, không có ảnh.

❓ Q4: "Tại sao dùng polling thay vì WebSocket?"
✅ Trả lời:
  WebSocket phù hợp khi cần kết nối liên tục 2 chiều (chat realtime,
  game, v.v.). Trong trường hợp này, em chỉ cần kiểm tra trạng thái
  một chiều (server → client) mỗi 5 giây.
  Polling đơn giản hơn, không cần cấu hình thêm, hoạt động tốt qua
  các lớp proxy/load balancer, và kết nối ngắn (request-response) nên
  không tốn tài nguyên giữ connection mở.
  Nếu đây là sản phẩm production với hàng nghìn user đồng thời,
  WebSocket hoặc Server-Sent Events sẽ hiệu quả hơn.

❓ Q5: "202 Accepted là gì? Sao không dùng 200?"
✅ Trả lời:
  HTTP 200 OK có nghĩa là request đã được xử lý xong và có kết quả.
  HTTP 202 Accepted có nghĩa là request đã được nhận và đang xử lý,
  kết quả sẽ có sau.
  Đây là convention chuẩn của REST API cho các tác vụ bất đồng bộ.
  Client nhìn thấy 202 biết ngay: "Tôi cần poll để lấy kết quả"
  thay vì nhầm tưởng request đã xử lý xong như 200.

❓ Q6: "AI có thể tạo nội dung không phù hợp không? Em xử lý thế nào?"
✅ Trả lời:
  Em xử lý ở 3 lớp:
    1. Input: giới hạn prompt 500 ký tự, tránh quá dài dễ bị lợi dụng.
    2. Prompt template: có câu cứng "Do not create 18+ adult content,
       violent, or disturbing image. Keep it family-friendly."
       Google Gemini cũng có safety filter riêng sẽ từ chối request vi phạm.
    3. Model level: Google Gemini có built-in content safety filters,
       nếu prompt vi phạm → model trả về error → backend cập nhật
       status = "failed" → người dùng thấy thông báo thất bại.

══════════════════ NHÓM 2: CÂU HỎI VỀ BACKEND & BẢO MẬT ══════════════════

❓ Q7: "Em bảo mật hệ thống như thế nào?"
✅ Trả lời:
  Em thực hiện bảo mật nhiều lớp:
  • Helmet: tự động security headers (ngăn XSS, clickjacking).
  • CORS: chỉ cho phép domain frontend cụ thể.
  • Rate limiting: 100 req/15 phút/IP → chống brute-force và DDoS nhỏ.
  • Firebase Auth token: mọi route nhạy cảm đều yêu cầu Bearer token,
    server verify bằng Firebase Admin SDK mỗi request.
  • Hai lớp admin auth: email whitelist (nhanh) + Firestore role (chính xác).
  • Input validation: Joi validate trước khi xử lý.
  • Chữ ký HMAC cho thanh toán: VNPay (SHA512), MoMo (SHA256).
    Sau callback, server tính lại chữ ký và so sánh → không thể giả mạo.
  • Ownership check: khi poll AI status, server kiểm tra userId trong
    Firestore phải khớp với uid trong token.

❓ Q8: "Tại sao dùng hai lớp Admin Auth?"
✅ Trả lời:
  Lớp 1 (auth.js): kiểm tra email có trong ADMIN_EMAILS list (array trong env).
  Nhanh, không cần query DB, dùng cho middleware chung.
  Lớp 2 (adminAuth.js): query Firestore, kiểm tra field role = "admin".
  Chính xác hơn, admin có thể được cấp/thu quyền linh hoạt qua DB.
  Hai lớp kết hợp: vừa nhanh vừa chính xác, vừa có thể scale.

❓ Q9: "HMAC là gì? Tại sao dùng nó cho thanh toán?"
✅ Trả lời:
  HMAC (Hash-based Message Authentication Code) là thuật toán tạo chữ
  ký số dựa trên nội dung dữ liệu + một secret key chỉ server biết.
  VNPay và MoMo yêu cầu mỗi request/callback phải kèm chữ ký HMAC.
  Khi callback về, backend tính lại HMAC từ dữ liệu nhận được với
  secret key → nếu khớp thì dữ liệu không bị giả mạo/chỉnh sửa.
  Điều này ngăn kẻ tấn công tự tạo callback giả để đánh dấu đơn hàng
  là "đã thanh toán" mà không thực sự trả tiền.

❓ Q10: "Firestore so với MySQL/PostgreSQL thế nào?"
✅ Trả lời:
  Firestore là NoSQL document database. Em chọn vì:
  • Tích hợp sẵn trong Firebase ecosystem (Auth, Storage = một nền tảng).
  • Không cần schema cứng → dễ thay đổi cấu trúc data trong quá trình dev.
  • Real-time listener hỗ trợ sẵn.
  • Free tier đủ cho đồ án.
  Nhược điểm: không hỗ trợ JOIN, truy vấn phức tạp khó; không có ACID
  như RDBMS; chi phí cao ở scale lớn.
  Nếu hệ thống có nhiều truy vấn phức tạp (báo cáo, thống kê), MySQL/
  PostgreSQL sẽ phù hợp hơn.

══════════════════ NHÓM 3: CÂU HỎI VỀ FRONTEND & KIẾN TRÚC ══════════════════

❓ Q11: "Tại sao dùng Zustand thay vì Redux?"
✅ Trả lời:
  Redux cần nhiều boilerplate: action, reducer, selector, middleware.
  Zustand API đơn giản hơn nhiều: tạo store 5-10 dòng code.
  Bundle size nhỏ hơn (~1KB vs ~7KB).
  Không cần Provider bao quanh toàn bộ app.
  Với quy mô đồ án này (3 store: auth, cart, favorite), Zustand là
  lựa chọn phù hợp. Nếu app lớn hơn với nhiều slice phức tạp,
  Redux Toolkit sẽ có ưu thế hơn trong cấu trúc và dev tools.

❓ Q12: "Next.js App Router khác Pages Router thế nào?"
✅ Trả lời:
  Pages Router (Next.js cũ): routing dựa trên file trong /pages,
  dùng getServerSideProps/getStaticProps cho data fetching.
  App Router (Next.js 13+): routing dựa trên /app, dùng React Server
  Components mặc định, hỗ trợ Layouts lồng nhau, loading.tsx,
  error.tsx tự động, fetch với cache granular.
  Em dùng App Router vì đây là hướng đi tương lai của Next.js,
  Layouts lồng nhau giúp chia Navbar/Footer không bị re-render.

❓ Q13: "SSR là gì? Em có dùng không?"
✅ Trả lời:
  SSR (Server-Side Rendering): HTML được render trên server trước
  rồi gửi về client → tốt cho SEO và first paint nhanh.
  Em dùng Next.js App Router, mặc định component là Server Component
  (render trên server). Các component cần tương tác người dùng
  (useState, useEffect) được đánh dấu "use client".
  Ví dụ trang Products list có thể fetch data server-side → HTML đầy
  đủ khi Google crawl → SEO tốt hơn SPA thuần.

❓ Q14: "Token hết hạn xử lý thế nào?"
✅ Trả lời:
  Firebase ID Token hết hạn sau 1 giờ. Em xử lý hai lớp:
  Lớp 1 (proactive): Axios request interceptor gọi getIdToken(true)
  trước mỗi request → Firebase SDK tự refresh token nếu gần hết hạn.
  Lớp 2 (reactive): Nếu server vẫn trả 401, Axios response interceptor
  bắt lỗi, thử refresh một lần nữa. Nếu vẫn lỗi → redirect /login.
  Người dùng hầu như không bao giờ thấy lỗi token hết hạn vì luôn
  dùng token mới nhất.

══════════════════ NHÓM 4: CÂU HỎI VỀ THIẾT KẾ & KINH DOANH ══════════════════

❓ Q15: "Điểm khác biệt của sản phẩm này so với website thương mại thông thường?"
✅ Trả lời:
  Điểm khác biệt chính là tính năng tạo tranh AI theo yêu cầu.
  Người dùng không chỉ mua tranh có sẵn mà có thể nhập bất kỳ ý tưởng
  nào → AI tạo ra tranh tô màu riêng biệt hoàn toàn cá nhân hóa.
  Kết hợp với chatbot biết sản phẩm thực của cửa hàng → trải nghiệm
  mua sắm thông minh và hiện đại hơn.

❓ Q16: "Hạn chế của hệ thống hiện tại và hướng cải thiện?"
✅ Trả lời:
  Hạn chế hiện tại:
    • AI tạo ảnh phụ thuộc vào Google API → nếu API thay đổi chính sách,
      hệ thống bị ảnh hưởng.
    • Polling mỗi 5 giây tạo nhiều request khi có nhiều user đồng thời
      → nên nâng cấp lên WebSocket hoặc Server-Sent Events.
    • Chưa có hệ thống review/rating sản phẩm.
    • Chưa có tính năng gợi ý sản phẩm dựa trên lịch sử mua.
  Hướng cải thiện:
    • Thêm SSE (Server-Sent Events) thay polling.
    • Thêm Redis cache cho product list để giảm Firestore reads.
    • Thêm tính năng customize thêm (chọn màu sắc palette cụ thể).
    • CI/CD pipeline để tự động deploy.

❓ Q17: "Em deploy hệ thống ở đâu?"
✅ Trả lời:
  Frontend: Vercel (có file vercel.json) → tự động CI/CD khi push code,
  hỗ trợ Next.js native, CDN toàn cầu, free tier tốt.
  Backend: Render (có render.yaml) hoặc Railway (có railway.json)
  → deploy Node.js Express app, free tier có giới hạn nhưng đủ demo.
  Database & Storage: Firebase (Firestore + Storage) → managed service,
  không cần tự quản lý server DB.

❓ Q18: "Em gặp khó khăn gì trong quá trình làm và giải quyết thế nào?"
✅ Gợi ý trả lời:
  Khó khăn 1: AI tạo ảnh lâu → timeout nếu dùng request đồng bộ.
    Giải pháp: Chuyển sang pattern 202 + async background + polling.

  Khó khăn 2: Token hết hạn trong quá trình dùng.
    Giải pháp: Axios interceptor tự refresh token trước mỗi request.

  Khó khăn 3: localStorage bị crash khi Next.js render phía server.
    Giải pháp: Tạo safeStorage wrapper kiểm tra typeof window trước.

  Khó khăn 4: Memory leak khi rời trang Generate giữa chừng polling.
    Giải pháp: Dùng useRef lưu intervalId, clearInterval trong cleanup.

  Khó khăn 5: Xác minh callback thanh toán để chống gian lận.
    Giải pháp: Tính lại HMAC từ dữ liệu nhận được, so sánh với chữ ký.
